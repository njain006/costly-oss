# Costly Connector Roadmap 2026

**Owner:** Nitin Jain
**Last updated:** 2026-04-23
**Audience:** YC W26 application review, design-partner prospects, OSS contributors
**Target:** Ship 20 additional connectors on top of the existing 16 to support the "25+ connectors — unified AI + data cost intelligence" pitch.

---

## 1. Executive Summary

Costly today ships 16 connectors covering Snowflake, AWS, BigQuery, Databricks, dbt Cloud, Fivetran, Airbyte, OpenAI, Anthropic, Claude Code, Gemini, Looker, Tableau, Omni, Monte Carlo, GitHub Actions, and GitLab CI. That is already a defensible slice of the modern data stack, but the YC W26 pitch hinges on two claims:

1. **Unified AI + data cost intelligence in one dashboard.** This requires closing the gap on the AI coding tools (Cursor, Copilot, Cohere, Bedrock) and the agentic inference layer (Together, Fireworks, Groq, OpenRouter, Hugging Face) — data teams today burn five-figure budgets on these tools and they are usually invisible to finance.
2. **A "single pane" for the whole data platform.** This requires filling the obvious gaps versus Vantage, Select, and Finout: Confluent/Kafka, ClickHouse, Hex, Dagster/Prefect, Azure OpenAI, and a reverse-ETL tool (Hightouch/Census). Data platform teams buying Costly today ask for these by name.

After ranking 40+ candidates on a 5-dimension impact score (user count, pain level, auth complexity, grouping fidelity, pitch resonance) and an effort score (S/M/L), the 20 highest (impact × ease) candidates cluster cleanly into three shipping waves:

- **Wave 1 — Pre-YC (ship by Sep 2026, ~10 connectors):** Cursor, GitHub Copilot, OpenRouter, Azure OpenAI, AWS Bedrock (deep view), Hex, Dagster Cloud + Prefect Cloud, Confluent Cloud, Hightouch, ClickHouse Cloud.
- **Wave 2 — Post-YC / pitch support (Q4 2026, 5 connectors):** Together AI, Fireworks AI, Pinecone, Vercel, Temporal Cloud.
- **Wave 3 — "Nice to have" for the 25+ badge (Q4 2026, 5 connectors):** Census, Astronomer, MotherDuck, Redpanda Cloud, Supabase.

Five explicit **defers** (too niche, duplicative, or hostile APIs) and two explicit **no-gos** (Datadog, Slack — costly should not compete there) round out the document.

## 2. Scoring Framework

Every candidate is scored on two composite axes.

**Impact (1–10):**
- User count / market relevance (0–3): How many data teams actually run this in production? Rough buckets: >50K teams = 3, 10K–50K = 2, 1K–10K = 1, <1K = 0.
- Cost-visibility pain level (0–3): How opaque is the bill? Does the vendor already show per-team cost inside their own UI? High opacity + surprise bills = 3.
- Pitch resonance for YC (0–2): Does shipping this connector sharpen the "AI + data cost" one-liner? Hero AI connectors get 2.
- Grouping fidelity (0–2): Does the API expose user/team/project/model dimensions, or only account totals? Fine-grained = 2.

**Effort:**
- **S** — < 1 day. REST endpoint with simple auth (bearer/API key) and a usage/cost endpoint that already returns normalized rows.
- **M** — 1–3 days. OAuth, pagination, multi-step token exchange, CSV export + parse, or usage/cost split across two endpoints.
- **L** — > 3 days. Non-trivial auth (mTLS, signed requests), GraphQL schemas, billing API gated on enterprise plan, or heavy normalization work.

Priority = Impact × (1/Effort weight). Ties broken by YC pitch resonance.

## 3. Wave 1 — Ship Before YC W26 (Target: 10 Connectors by Sep 2026)

These ten are the minimum viable set to plausibly claim "unified AI + data cost intelligence" in the W26 application and front-page the live demo.

### 3.1 Cursor (AI coding)

- **Category:** AI / coding assistant
- **User count:** ~1M+ paid seats across startups/enterprise by 2026 ([Vantage: Cursor Pricing Explained](https://www.vantage.sh/blog/cursor-pricing-explained)); one of the highest-velocity dev-tool purchases of the last 24 months.
- **Pain level:** 3/3. Cursor moved to a credit-based model in June 2025 where the monthly credit pool depletes based on *which* model the developer picks (Opus vs Sonnet vs Haiku vs GPT-5). Finance teams have no intuition for this and bills routinely triple month over month ([Cursor Docs: Models & Pricing](https://cursor.com/docs/models-and-pricing)).
- **Canonical data source:** Admin Dashboard API (Enterprise); Teams plan exposes per-user usage via CSV export. The enterprise "AI code tracking API and audit logs" entitlement is the right hook ([Cursor Business plan](https://cursor.com/docs/models-and-pricing)).
- **Auth:** API key generated by Team/Enterprise admin.
- **Grouping:** user, model, date, project (if tagged).
- **Pricing tier structure:** $20/mo Pro, $40/seat/mo Business, custom Enterprise. Credit pool = plan price in dollars.
- **Effort:** M. Admin API is gated behind Enterprise today; Teams pilots will start with CSV import until the Team-tier API lands.
- **Why it belongs:** Cursor is the single most-requested AI connector from every design-partner call. If the YC demo cannot show per-engineer Cursor spend alongside Anthropic API spend, the "unified" claim is empty.

### 3.2 GitHub Copilot Business / Enterprise (AI coding)

- **Category:** AI / coding assistant
- **User count:** 1.8M+ paid Business/Enterprise seats; the default corporate choice.
- **Pain level:** 2/3. Per-seat pricing is predictable, but adoption attribution and ROI are not — finance wants to know who uses their seat and who doesn't.
- **Canonical data source:** `GET /enterprises/{enterprise}/copilot/metrics` and `GET /orgs/{org}/copilot/metrics` — GA as of Feb 27, 2026 ([GitHub Changelog: Copilot metrics GA](https://github.blog/changelog/2026-02-27-copilot-metrics-is-now-generally-available/)). Also `GET /enterprises/{enterprise}/copilot/usage` for the 28-day user-level report ([REST API: Copilot usage metrics](https://docs.github.com/en/rest/copilot/copilot-usage-metrics)).
- **Auth:** GitHub PAT with `manage_billing:copilot` or `read:enterprise` scope.
- **Grouping:** user, team, editor, feature (chat/cli/code-review), language, day.
- **Pricing:** $19/seat/mo Business, $39/seat/mo Enterprise (both annual billing).
- **Effort:** S. Reuse the existing GitHub connector auth layer; add a Copilot-metrics service module.
- **Why it belongs:** Lets Costly surface "paid Copilot seats with <5 accepts/week" — an immediate, quantifiable savings recommendation we can put in the pitch deck.

### 3.3 OpenRouter (AI router / inference gateway)

- **Category:** AI / LLM inference gateway
- **User count:** Fast-growing — 300+ models; default choice for multi-model agentic workloads.
- **Pain level:** 3/3. Teams route 10+ providers through OpenRouter and lose per-model cost visibility inside their own observability stack. OpenRouter itself ships the primitives to solve this, which is why this is an S-tier connector for us.
- **Canonical data source:** Activity tab + per-request `total_cost` and `usage` fields in every API response ([OpenRouter: Usage Accounting docs](https://openrouter.ai/docs/guides/administration/usage-accounting)).
- **Auth:** API key (provisioning-scoped keys supported).
- **Grouping:** model, provider, API key (proxies to "team" / "agent"), request, day.
- **Pricing:** Pass-through provider pricing + a small platform fee; no markup on inference ([OpenRouter Pricing](https://openrouter.ai/pricing)).
- **Effort:** S. Well-documented REST API, per-request cost already pre-computed.
- **Why it belongs:** OpenRouter is the most "costly-shaped" vendor on the list — the API returns exactly the normalized rows we need. This connector alone unlocks "one dashboard for 300 LLM models."

### 3.4 Azure OpenAI (AI / LLM)

- **Category:** AI / LLM (hyperscaler-hosted)
- **User count:** 60%+ of Fortune 500 AI deployments; anywhere that can't ship to OpenAI directly for compliance reasons.
- **Pain level:** 3/3. Billing is split across Standard (per-token), PTU (per-hour reservation), and Batch (50% off). Token meters appear *per model, per deployment* in Azure Cost Management, which is painful to reconcile ([Azure OpenAI cost management](https://learn.microsoft.com/en-us/azure/ai-services/openai/how-to/manage-costs)).
- **Canonical data source:** Azure Cost Management API (Consumption/UsageDetails) filtered to `Microsoft.CognitiveServices/accounts` with `meterCategory = "Azure OpenAI"`.
- **Auth:** Azure AD service principal with Cost Management Reader.
- **Grouping:** subscription, resource group, deployment name, model, meter (input/output/training/hosting), region, day.
- **Pricing:** On-demand token pricing + PTU reservations + fine-tuning hosting. Batch is 50% discount.
- **Effort:** M. Azure auth is heavier than AWS (tenant/client/secret triple), and filtering the meter hierarchy to only Azure OpenAI requires care.
- **Why it belongs:** Closes the "all three hyperscaler AI services" loop (OpenAI + AWS Bedrock + Azure OpenAI). Enterprises running on Azure literally cannot use Costly today without this.

### 3.5 AWS Bedrock (Deep view on top of the AWS connector)

- **Category:** AI / LLM (hyperscaler-hosted)
- **User count:** Largest enterprise foundation-model deployment by volume; default for any AWS-first shop.
- **Pain level:** 3/3. Costs are driven by input/output/cache-read/cache-write tokens priced separately, with 5-minute billing windows for on-demand and 24–48h billing lag ([AWS Bedrock pricing](https://aws.amazon.com/bedrock/pricing/)). Without application inference profiles, cost-by-team is invisible.
- **Canonical data source:** Cost Explorer (already wired) + new granular cost attribution released in 2025 that attributes inference cost to the IAM principal that made the call, accessible via CUR 2.0 ([AWS: Granular cost attribution for Bedrock](https://aws.amazon.com/blogs/machine-learning/introducing-granular-cost-attribution-for-amazon-bedrock/)). Also supports application inference profiles for app-level tagging.
- **Auth:** Reuse existing AWS IAM role; add `cur:DescribeReportDefinitions` and CUR S3 read.
- **Grouping:** model, region, IAM principal, application inference profile, input/output/cache token type, day.
- **Pricing:** Per-token on-demand, provisioned throughput, and model-copy hourly.
- **Effort:** M. Bedrock lives inside the existing AWS connector but deserves a dedicated service-level view because the drill-downs (model, principal, token type) are different from EC2/S3.
- **Why it belongs:** Publicly branding "costly is Bedrock-native" is a better wedge than "costly is AWS-native" — the AWS market is owned by Vantage/ProsperOps, but the Bedrock FinOps niche is wide open.

### 3.6 Hex (AI-native BI / notebook)

- **Category:** BI / analytics / notebook
- **User count:** 2,000+ paying teams, explosive growth among modern data teams — the new Mode.
- **Pain level:** 3/3. Hex compute bills by the minute across tiered compute profiles, and credits for "Hex Agents" (AI features) are billed separately and *very* opaquely ([Hex: Compute docs](https://learn.hex.tech/docs/administration/workspace_settings/compute), [Hex pricing](https://hex.tech/pricing/)). Admins can cap per-cycle compute but cannot natively chargeback to teams.
- **Canonical data source:** Hex admin REST API (audit events + project metadata) + workspace compute-usage endpoint (available on Team/Enterprise).
- **Auth:** Workspace-scoped personal access token.
- **Grouping:** project, creator, compute profile, run type (scheduled/manual), day.
- **Pricing:** Team $149–$199/Creator seat/mo + $100 compute billing buckets; Enterprise custom.
- **Effort:** M. Need to stitch together project metadata + compute usage + agent credit usage.
- **Why it belongs:** Hex is the BI tool most over-represented in data-team Slack channels in 2026 — "our Hex bill doubled" is a Tier-0 pain point. Shipping Hex alongside Looker/Tableau/Omni gives Costly the full modern BI spread.

### 3.7 Dagster Cloud + Prefect Cloud (Orchestration, bundled)

- **Category:** Orchestration
- **User count:** Dagster ~5K teams, Prefect ~8K teams. Airflow (self-hosted) is larger but attribution requires a different approach (see §6).
- **Pain level:** 3/3 (Dagster), 2/3 (Prefect). Dagster switched to a pure pay-as-you-go credit model effective May 1, 2026 — every credit is billed from zero at $0.035–$0.040 each on top of the same base fee ([Dagster pricing](https://dagster.io/pricing)). Teams that planned against the old "included credits" model are experiencing 2–3x bill jumps right now. Prefect's seat + serverless-overage model ($0.005/min) is simpler but still worth tracking ([Prefect vs Dagster self-serve plans](https://www.prefect.io/blog/dagster-vs-prefect-self-serve-plans-compared)).
- **Canonical data source:**
  - Dagster: Cost Insights API (per-asset credit usage, duration, trigger type) — explicitly designed for chargeback ([Dagster Cost Insights](https://dagster.io/platform-overview/cost-insights)).
  - Prefect: Workspaces API + usage endpoint for serverless minutes.
- **Auth:** Dagster: deployment-scoped user token. Prefect: service account API key.
- **Grouping:** asset/flow, deployment, run type, day, trigger.
- **Pricing:**
  - Dagster Solo $10/mo + $0.040/credit, Starter $100/mo + $0.035/credit, Serverless $0.010/min.
  - Prefect seat-based + serverless-minute overage.
- **Effort:** M for Dagster (the asset-level cost insights endpoint returns dense structured data), M for Prefect.
- **Why it belongs:** Data orchestration is the single most cited "we don't know what this costs" category in FinOps surveys — because runs are scheduled and untagged, nobody knows what the nightly workload actually costs. Bundled together, these two cover 70% of the modern orchestration market (Airflow self-hosted is a separate §6 item).

### 3.8 Confluent Cloud / Kafka (Streaming)

- **Category:** Streaming / messaging
- **User count:** 4,500+ paying customers, default managed Kafka.
- **Pain level:** 3/3. Confluent bills across 15+ dimensions (eCKU, ingress/egress GB, storage GB, partitions, connectors, ksqlDB, Schema Registry) and teams routinely misunderstand the *Dedicated* vs *Basic* vs *Standard* cost curves ([Confluent Billing Overview](https://docs.confluent.io/cloud/current/billing/overview.html), [Consumption Metrics](https://docs.confluent.io/cloud/current/billing/units.html)).
- **Canonical data source:** Confluent Cost API (`GET /billing/v1/costs`) — returns one line item per billing dimension with unit of measure, supports date-range queries ([Confluent billing docs](https://docs.confluent.io/cloud/current/billing/overview.html)). Metrics API with `principal_id` label enables per-team chargeback on Dedicated ([Metrics by principal](https://docs.confluent.io/cloud/current/monitoring/metrics-by-principal.html)).
- **Auth:** Confluent Cloud API key + secret (BasicAuth).
- **Grouping:** environment, cluster, billing dimension, principal (team), day.
- **Pricing:** eCKU-based + per-GB transfer + storage. Basic/Standard/Enterprise/Dedicated tiers.
- **Effort:** M. The cost endpoint lags by 72 hours which has to be surfaced in the UI; pairing with the Metrics API for same-day estimates is a useful UX touch.
- **Why it belongs:** Kafka cost management is a greenfield — Vantage only added it in 2024 and still no one does it well for teams that run their own workloads. Shipping this alongside Snowflake + Databricks + BigQuery makes Costly the first tool with real-time *and* streaming platform cost coverage.

### 3.9 Hightouch (Reverse ETL / data activation)

- **Category:** Ingestion / reverse ETL / activation
- **User count:** ~2K paying teams, category-leader alongside Census.
- **Pain level:** 2/3. Hightouch bills on "active destinations" with no MTU caps — simpler than Census but still opaque to finance because every destination sync triggers downstream SaaS cost (HubSpot/Salesforce usage) that's invisible ([Hightouch Pricing](https://hightouch.com/pricing)).
- **Canonical data source:** Hightouch REST API — workspaces, sources, destinations, syncs, and usage endpoints ([Hightouch API docs](https://hightouch.com/docs/api)).
- **Auth:** Workspace API key.
- **Grouping:** workspace, source, destination, sync, model, day.
- **Pricing:** Starter free tier + $350/mo; Business custom.
- **Effort:** S–M. REST API is clean; destinations-based billing needs mapping to UnifiedCost rows.
- **Why it belongs:** Reverse ETL is the hot category Costly currently has zero coverage of. Hightouch picked over Census for Wave 1 because (a) the API is better-documented and (b) Hightouch ships more volume. Census lands in Wave 3.

### 3.10 ClickHouse Cloud (Warehouse / lakehouse)

- **Category:** Warehouse / analytic DB
- **User count:** 2,000+ paying clusters, fastest-growing warehouse alternative in 2026.
- **Pain level:** 3/3. Compute scales based on concurrent queries on Scale tier, storage on Enterprise tier; dimensions are numerous and non-obvious.
- **Canonical data source:** ClickHouse Organization `usageCost` API — returns usage + cost by subcategory, service, tags ([ClickHouse Billing API announcement](https://clickhouse.com/blog/announcing-billing-api-for-clickhouse-cloud-with-vantage-support)).
- **Auth:** ClickHouse Cloud API key (Developer role is enough for read-only).
- **Grouping:** organization, service, cluster, subcategory (compute/storage/transfer), tag, day.
- **Pricing:** Development/Production/Enterprise tiers + compute-hours + storage-GB + transfer.
- **Effort:** S. Vantage has already reverse-engineered this API and ClickHouse publishes it officially — one of the cleanest billing APIs in the industry.
- **Why it belongs:** ClickHouse is the "third leg" of the warehouse stool after Snowflake and BigQuery for analytics-heavy shops. A credible data-platform pitch in 2026 has all three.

## 4. Wave 2 — Q4 2026 "Fills out the 25+ pitch" (5 Connectors)

These are genuinely strong connectors but either (a) the API is harder, (b) the user base is narrower, or (c) they're already indirectly covered through a sibling connector.

### 4.1 Together AI + Fireworks AI (Bundled AI inference)

- **Category:** AI / inference
- **User count:** Together ~200K devs, Fireworks ~150K devs; both growing fast in the OSS model deployment niche.
- **Pain level:** 2/3. Both ship dashboards, but both are aggregate-only (no per-request breakdown on Fireworks) and neither surfaces multi-provider views ([Clarifai inference comparison](https://www.clarifai.com/blog/clarifai-vs-other-inference-providers)).
- **Canonical data source:** Together AI usage endpoint (per-model, per-key); Fireworks usage endpoint (per-model, aggregate).
- **Auth:** API key.
- **Grouping:** model, API key, day. (Fireworks is weaker here.)
- **Effort:** S each. Bundled because they share OpenAI-compatible endpoints — one connector spec, two auth adapters.
- **Why it belongs:** Pitch moment: "Costly tracks the big three (OpenAI, Anthropic, Gemini) *and* every major open-weights inference provider (Together, Fireworks, Groq via OpenRouter)." That is a concrete line no competitor can say.

### 4.2 Pinecone (Vector DB)

- **Category:** Vector database / ML serving
- **User count:** ~50K paid workspaces, category leader.
- **Pain level:** 2/3. Serverless read-unit/write-unit/storage pricing is opaque; $50 monthly minimum surprises hobbyists ([Pinecone Pricing](https://www.pinecone.io/pricing/), [Pinecone cost guide](https://docs.pinecone.io/guides/manage-cost/understanding-cost)).
- **Canonical data source:** Pinecone management API (`/indexes`, `/usage`), billing on Enterprise tier.
- **Auth:** API key.
- **Grouping:** index, namespace, day.
- **Effort:** M. The usage endpoint is newer and Enterprise-gated.
- **Why it belongs:** Every RAG-stack design partner mentions Pinecone. Bundled with OpenAI + Anthropic it tells a complete "RAG cost" story.

### 4.3 Vercel (Edge / frontend platform)

- **Category:** Infrastructure / edge
- **User count:** ~2M developers, largest frontend host.
- **Pain level:** 3/3. Historically a black box; now much better.
- **Canonical data source:** `/billing/charges` endpoint — returns FOCUS v1.3 newline-delimited JSON, up to 1-year date range, 1-day granularity ([Vercel Changelog: billing API](https://vercel.com/changelog/access-billing-usage-cost-data-api)).
- **Auth:** Team-scoped API token.
- **Grouping:** team, project, service (Functions/KV/Postgres/Blob/AI Gateway), day.
- **Effort:** S. FOCUS-compliant output means our normalizer is almost trivial.
- **Why it belongs:** Frontend platforms are the "shadow data stack" that finance never sees. Vercel + Cloudflare + Supabase cover most early-stage AI apps. Vercel goes in Wave 2 because they're largest and the API is FOCUS-native.

### 4.4 Temporal Cloud (Orchestration / workflow)

- **Category:** Orchestration / workflow engine
- **User count:** ~1,500 paying organizations, but each burns significant Actions volume.
- **Pain level:** 3/3. Billing is based on "Actions" ($50 per million) + active storage + retention — three orthogonal dimensions ([Temporal Pricing](https://docs.temporal.io/cloud/pricing)).
- **Canonical data source:** Temporal Cloud Billing API (Pre-release) — CSV reports at namespace granularity, hourly for current and previous billing month ([Temporal Cloud Billing API](https://docs.temporal.io/cloud/billing-api)).
- **Auth:** Cloud-level API key.
- **Grouping:** account, namespace, Action type, day.
- **Effort:** M. API is pre-release (watch for breaking changes).
- **Why it belongs:** Temporal replaces Airflow for event-driven workloads at high-performance shops (Stripe, Coinbase, etc.). Costly supporting it is a direct wedge into that segment.

### 4.5 Snowflake-adjacent / Warehouse: Redshift as an AWS sub-service

Not technically a new connector — **this is a deep-view enhancement to the AWS connector** similar to the AWS Bedrock treatment in 3.5. Ship it in Wave 2 because Redshift is still a significant line item for AWS-native data teams and the existing AWS connector already treats it as a single SKU. Effort S, impact 5/10.

(Redshift got bumped out of its own slot because its API and billing live entirely inside CUR — treating it as a sub-service under AWS is simpler.)

## 5. Wave 3 — Q4 2026 "Pitch completeness to 25+" (5 Connectors)

### 5.1 Census (Reverse ETL)

Shipped second behind Hightouch for the reverse-ETL coverage story ([Hightouch vs Census](https://hightouch.com/blog/hightouch-vs-census)). Pricing model (destination-fields) is harder to normalize but we already know the shape from Hightouch. Effort S–M. Census pairs with Hightouch for the "full reverse ETL market" line.

### 5.2 Astronomer (Managed Airflow)

- Category: Orchestration / Airflow.
- User count: ~700 paying orgs.
- Pain level: 2/3. Astro bills on worker hours + scheduler hours + data transfer ([Astronomer Pricing](https://www.astronomer.io/pricing/)); the per-deployment metrics export is Prometheus-remote-write format ([Astronomer Export Metrics](https://www.astronomer.io/docs/astro/export-metrics)).
- Effort: M (Prometheus scraping, not REST).

### 5.3 MotherDuck (DuckDB SaaS warehouse)

- Category: Warehouse.
- Pain level: 2/3. Lite $25/mo + Business $100/mo + $0.08/GB storage + Compute Unit-seconds billed per-instance-type ([MotherDuck Pricing](https://motherduck.com/product/pricing/)). Niche but trendy.
- Effort: S. Clean API, small surface area.
- Why bundle into Wave 3: It's a hero Show-HN talking point ("we track DuckDB SaaS cost!") without being a big business driver.

### 5.4 Redpanda Cloud (Streaming)

- Category: Streaming, direct Kafka competitor.
- Pain level: 2/3. Less complex billing than Confluent, CSV download available ([Redpanda Billing docs](https://docs.redpanda.com/redpanda-cloud/billing/billing/)).
- Effort: M. No first-class cost API today; CSV ingestion required. Worth it for the completeness message.

### 5.5 Supabase (Backend-as-a-service / Postgres)

- Category: Serverless DB + auth + storage.
- User count: 1M+ projects, default hobby/small-startup Postgres.
- Pain level: 2/3. Pro $25/mo/project then per-usage. Metrics API is Prometheus-compatible ([Supabase Metrics API](https://supabase.com/docs/guides/telemetry/metrics)); billing API is weaker.
- Effort: M. Metrics scrape + billing event import.

**Result:** Shipping 5.1–5.5 pushes Costly from 16 → 25 → 30+ connectors by end of Q4 2026 (depending on what gets pulled forward).

## 6. Explicit Defers (5)

These are genuinely interesting but don't make the first 20. Each gets a one-paragraph rationale so we can defend the "why not" in OSS issues.

### 6.1 Airflow (Self-hosted)

Self-hosted Airflow has no native billing — cost is the sum of underlying compute (EKS/EC2/GKE). Attribution should be done *through* the warehouse/cloud connector using DAG tags, not by instrumenting Airflow itself. Right implementation is a **"cost attribution rule"** feature in Costly that lets users map compute to DAGs — not a connector. Defer until the attribution-rules feature ships.

### 6.2 Weaviate + Qdrant Cloud

Already covered by Pinecone for the pitch. Both have <1/10th the paying users of Pinecone ([Vector DB Pricing Comparison 2026](https://ranksquire.com/2026/03/04/vector-database-pricing-comparison-2026/)) and neither has a mature billing API. Revisit in 2027 if the OSS community clamors for it.

### 6.3 Coalesce, Prophecy, SQLMesh Cloud

All three are transform-layer competitors to dbt Cloud (which we already cover). User count is small (each <2K paid teams), billing APIs are immature, and the differentiation to finance-facing buyers is minimal — they all look like dbt bills. Revisit when one crosses 10K teams.

### 6.4 Great Expectations Cloud, Soda Cloud, Elementary Cloud, Bigeye, SYNQ

Data-quality tools. Monte Carlo already covers the category for us. Adding a second data-quality connector before we have a second AI coding tool (Cursor), a second orchestrator (Dagster), or a second reverse-ETL tool (Hightouch) is wrong sequencing. Wave 4 (2027) if design partners ask.

### 6.5 Tecton, Hopsworks, Featureform (Feature stores)

Feature stores are a genuine ML-platform primitive but the market is <500 paying teams total. Wait for a paying design partner to specifically request this.

## 7. Explicit No-Gos (The "We Shouldn't Build These" List)

Costly should not attempt to compete head-on in these markets — they own their segment, the moat is 5+ years of data, and the integration effort is a net negative.

### 7.1 Datadog (itself)

Datadog is a $50B+ public company whose *entire* business is observability-and-cost-of-observability. Their own CCM product (Cloud Cost Management) is direct competition for Costly's AWS/Azure/GCP coverage. Asking "what does Datadog cost?" inside Costly is a useful feature, but it requires importing Datadog's own usage CSV which Datadog will deprecate at will. **Defer indefinitely**; if a user wants Datadog cost analysis, the right answer is "use Datadog CCM." ([Datadog Cost Attribution](https://docs.datadoghq.com/account_management/billing/usage_attribution/)).

### 7.2 Slack / Notion / Linear (Collaboration SaaS)

Outside Costly's "data + AI platform" wheelhouse. Finance teams already have Vendr, Torii, and Zylo for general SaaS spend management. Integrating Slack seat cost adds noise to the dashboard and muddies the pitch. **Never build these.**

### 7.3 Sentry, New Relic, Grafana Cloud (APM)

Same reasoning as Datadog. APM is its own category with entrenched players and their own cost dashboards. ([Sentry Spend Allocation](https://docs.sentry.io/pricing/quotas/spend-allocation/)). Might revisit Grafana Cloud as a *source* of metrics for Costly (dashboard export), but not as a cost-connector target.

### 7.4 Perplexity for Teams

Perplexity Team ($40/seat/mo) has no usage API and no meaningful per-seat variance — it's a SaaS license. Not a good fit for Costly's cost-attribution muscle. Revisit only if Perplexity exposes per-query token cost in 2027.

### 7.5 Hyperscaler ML platforms as separate connectors (SageMaker, Vertex AI Model Garden)

Same call as Bedrock — these are *sub-services* of AWS/GCP and should be deep-views inside the existing AWS and BigQuery/Vertex connectors, not standalone connectors. Building them as separate connectors would be double-counting in the "25+" claim, which is a pitch risk if reviewers notice.

## 10. Open Questions / Risks

1. **Cursor Admin API gating.** Today the detailed API is Enterprise-only; Teams gets CSV export. If Cursor doesn't ship a Teams API by Q3, we need to pivot to a CSV-import adapter (acceptable but loses the "live dashboard" feel).
2. **Temporal Billing API is pre-release.** Breaking changes are possible. Acceptable risk — Temporal users are technical and will tolerate instability.
3. **AWS Bedrock application inference profiles** are the attribution unlock, but require customer setup work. Our onboarding flow needs to walk customers through tagging.
4. **Azure auth complexity.** Azure service-principal setup is historically a friction point. Plan for a "walkthrough wizard" in the UI similar to the existing Snowflake key-pair setup.
5. **Hex compute reporting lives behind Team+ plans.** Free/Starter workspaces won't have the API we need. Segment the marketing copy accordingly.

## 11. Appendix — Why the "25+" Claim Matters for YC

The W26 pitch deck lives or dies on three pattern-matching cues:

1. **"AI + data in one place"** — The reviewer's literal first question is "why are these two things in the same product?" The answer must be reflex-fast: because the same team pays for both, the same engineer uses both, and nobody shows both in one chart today. Connectors #1–7 above are what make this true on the screen.

2. **"Real connectors, not slideware"** — Every YC reviewer has been burned by pitches claiming "integrations" that are screenshots. 25+ *live* connectors with real customer data visible in the demo is the trust-establishing move. Wave 1 gets us to 26; Wave 2 gets us to 31.

3. **"Why can't Vantage do this?"** — Vantage is an 8-year-old FinOps company with 50+ connectors but (a) zero AI coding tools, (b) no BI tools, and (c) no orchestration. By shipping Cursor + Copilot + Hex + Dagster + Prefect in Wave 1, Costly occupies a distinct corner that Vantage cannot easily enter without cannibalizing their cloud-FinOps positioning.
